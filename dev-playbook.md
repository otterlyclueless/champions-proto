# 🧩 Issue Log — Mobile Teams Layout (April 2026) - Session 1

## ❌ Problem
- Teams detail view looked broken on mobile
- Cards stayed in desktop-style horizontal layout
- CSS changes appeared to have no effect

---

## 🔍 What We Tried (That Didn’t Work)
- Added mobile CSS targeting:
  - `.member-card`
  - `.member-main`
  - `.member-right`
  - `.team-detail-grid`
- Forced stacking layouts using flex/grid
- Adjusted spacing, buttons, and card styles

Result:
→ No visible change in UI

---

## 🧠 Root Cause

### 1. CSS Was Targeting Assumed Classes
- The class names used in CSS were not confirmed
- Teams UI is not defined in `index.html`

### 2. Teams Layout is Dynamically Rendered
```html
<div id="teamsView"></div>
```
- All team cards are injected via `app.js`
- Real structure only exists in the rendered DOM

### 3. We Were Styling Without Inspecting the DOM
- CSS was written based on guesses
- Not based on actual rendered markup

---

## 💡 Key Learnings

### 🧭 The DOM is the Source of Truth
- What you see in the browser = reality
- Not what’s in your files

---

### 🎯 CSS Must Match Real Selectors
- If a class does not exist in the DOM → CSS does nothing

---

### 🔍 Always Use DevTools → Elements
- Inspect real structure
- Confirm class names
- Check applied styles

---

### 📱 Mobile Layout Requires Structural Change
- Desktop layout ≠ mobile layout
- Don’t squeeze horizontal layouts
- Stack vertically instead

---

### ⚠️ “Nothing Changed” = Debug Signal
Usually means:
- Wrong selector
- Dynamic HTML mismatch
- CSS not targeting real elements

---

## 🛠️ Correct Debug Approach (Pattern)

When UI changes don’t apply:

1. Open DevTools → Elements
2. Inspect the element
3. Identify real class names
4. Map DOM structure
5. Write CSS based on reality
6. Test immediately

---

## 🧱 Architecture Reminder

| Layer        | Responsibility              |
|--------------|-----------------------------|
| `index.html` | Static layout               |
| `app.js`     | Dynamic UI rendering        |
| `styles.css` | Visual styling              |

Teams view lives in `app.js`, not `index.html`

---

## 🧠 Core Principle

> Don’t guess the UI — inspect it.

> DOM → CSS → UI

Not:

> CSS → Guess → Hope

---

## 🚀 Next Step (Follow-up Work)

- Identify real class names
- Rebuild mobile layout using correct selectors

## 🧩 Issue: Mobile Teams Layout Not Updating (April 2026) - Session 2

### ❌ Problem

- Teams detail view appeared broken on mobile  
- Layout remained horizontal (desktop-style) instead of stacking  
- UI felt cramped and unreadable on smaller screens  
- CSS changes had no visible effect  

Expected:  
→ Clean vertical stacking layout for mobile  

Actual:  
→ No UI change despite multiple CSS updates  

---

### 🔍 What We Tried

- Added mobile CSS targeting:
  - `.member-card`
  - `.member-main`
  - `.member-right`
  - `.team-detail-grid`
- Forced stacking using flex and grid overrides  
- Adjusted spacing, buttons, and layout rules  

Result:

→ No visible change in UI  

---

### 🧠 Root Cause

#### 1. Incorrect Selector Assumptions

- CSS targeted class names that were not confirmed in the DOM  
- Styles applied to non-existent elements → no effect  

#### 2. Dynamic Rendering Misunderstood

- Teams UI is rendered via JavaScript (`app.js`)  
- Not present in static `index.html`  
- Structure only exists at runtime  

#### 3. No DOM Inspection Step

- CSS was written based on assumptions  
- DevTools inspection was skipped  
- Real structure was never verified  

---

### 💡 Key Learnings

- The DOM is the single source of truth for UI structure  
- CSS must target **real, rendered selectors**, not assumed ones  
- Dynamically rendered UI requires inspection after render  
- “No visual change” is a **diagnostic signal**, not a dead end  
- Mobile layout fixes often require **structural changes**, not just styling tweaks  

---

### 🛠️ Pattern (Reusable Solution)

1. Open DevTools → Elements  
2. Inspect the actual rendered component  
3. Identify real class names and hierarchy  
4. Map the DOM structure  
5. Write CSS against confirmed selectors  
6. Test changes immediately in DevTools before committing  

---

### 🧱 Architecture Notes

| Layer        | Responsibility              |
|--------------|-----------------------------|
| index.html   | Static layout               |
| app.js       | Dynamic UI rendering        |
| styles.css   | Visual styling              |

Teams view is injected into:

<div id="teamsView"></div>

---

### 🚀 Resolution

- Inspected rendered DOM using DevTools  
- Identified actual class structure generated by `app.js`  
- Rewrote mobile CSS using correct selectors  
- Shifted layout approach from “override desktop” → “design for mobile structure”

## 🧩 Drop E — Move Picker (April 2026)

### 💡 Lessons

**Two-layer shell/body render preserves input focus**
- Following Drop D.2's pattern: `renderMovePickerShell()` writes the sticky chrome (search input, tabs) ONCE per open; `renderMovePickerBody()` re-renders just the list on every search/filter keystroke.
- If you re-render the shell on input, the `<input>` element is destroyed mid-keystroke and focus is lost → broken UX.

**Dynamic filter pills should reflect REAL data**
- Don't show all 18 Pokémon types in the filter row. Only show types present in the CURRENT species' learnset.
- `mpLegalTypes()` dedupes types from `_mpLearnset` and returns them sorted.

**Hidden input contract keeps saveBuild() unchanged**
- Builds save `move_1..move_4` as TEXT move names. The picker writes to hidden `<input>` elements (`edM1..edM4`) and `saveBuild()` reads those — zero migration.

**Script load order matters across modules**
- `app-builds.js` references `displayName()` (from `app-pokedex.js`) and `allMoveIndex` (from `app-core.js`). Pokedex loads before builds so `displayName` is defined at user-interaction time. ✅

**Per-species cache is the right granularity**
- `learnsetCache[pokemonId]` avoids re-querying Supabase when users re-open the picker for the same Pokémon. 258 distinct species, 63.8 avg moves each = modest memory footprint.

**allMoveIndex powers more than the picker**
- The same lookup (populated once at boot) drives the build-detail move cards with type-matched gradients, and the editor slot buttons. One load, three UIs.

### 🧭 DOM-truth reminder (still true)

- The move picker overlay is a new `<div id="movePickerOv">` added in `index.html`. Styles target `.mp-*` namespace to avoid collisions with Drop D.2's `.picker-item`/`.picker-sheet`/`.picker-head` classes (which the item picker still uses).

### 🧹 Housekeeping debt

- `app-builds.js` is now ~72KB. Both the item picker (Drop D.2) and the move picker (Drop E) live here. A future drop could extract them into a shared `app-pickers.js` module with a common base — worth it if a third picker (ability, nature) is added.

# 🔀 Drop F.1 — Hash Router + Share Code Foundation (April 22 2026)

## ✅ What shipped

- **`app/app-router.js`** (new module, ~220 lines) — hash parser, share-code generator, stub public renderers. Loaded between view modules and `app-init.js`.
- **`parseHash(hash)`** — returns `{kind:'build'|'team'|'profile', code|name}` or `null`. Strict regex: builds/teams accept 4-8 char base62, usernames 3-20 chars `[a-zA-Z0-9_-]` and are lowercased.
- **`genShareCode(len=5)`** — base62 via `crypto.getRandomValues`. 916M combos at 5 chars. Falls back to `Math.random` on ancient browsers.
- **`ensureShareCode(kind, id)`** — fetches current `share_code`, if null generates + PATCHes with retry. Catches PostgREST 409 or body `code: '23505'` (Postgres unique violation) and retries up to 8 times.
- **Stub public renderers** — minimal card: sprite + type chips + name + `by @username` byline + meta strip. Signed-out visitors see a sticky red "Create your build →" FAB; signed-in viewers see "Back to my app".
- **`index.html`** — added `<div class="page" id="pg-public"></div>` + router script tag.
- **`app-init.js`** — detects public route on first load; suppresses `maybeShowInitialAuthPrompt()` when on one; calls `_championsRouter.init()` after bootstrap.
- **`styles.css`** — `.pub-*` class set. When `body.pub-route:not(.pub-route-authed)`, the sidebar hides entirely (signed-out visitors have no nav target anyway).

## 🧪 End-to-end test (against live DB)

Temporarily set Aaron's Mimikyu build to `is_public=true, share_code='t3st0'` via service_role, then fetched via the **anon key only** (simulating a signed-out visitor):

- ✅ `builds?share_code=eq.t3st0&is_public=eq.true` → returned the row (RLS policy "Anon read public or in-public-team builds" works)
- ✅ `user_profiles?user_id=eq.<aaron-uid>` → returned `{username, display_name}` (RLS policy "Read public content authors" works — anon can read profiles of users who have any public content)
- ✅ `pokemon?id=eq.<mimikyu-uid>` → returned full row (no RLS on pokemon)
- ✅ Wrong share_code → empty array (no leakage)
- ✅ After revert to private → anon immediately sees empty array again

## 🔑 Key decisions

- **Short codes, fixed 5 chars** — matches the handover brief's example `a8k2p`. 916M combos is overkill for current scale but leaves plenty of headroom without making URLs ugly.
- **New module, not app-core extension** — routing is cross-cutting enough to warrant its own file. `app-builds.js` is already 74KB and `app-core.js` is 22KB; splitting router out keeps each module focused.
- **Stub renderer included** — a bare router with no visible output is hard to test. Including minimal HTML proves the full round-trip (hash → fetch → RLS → render) works before we invest in the full Drop F.2 read-only detail layouts.
- **No share toggle yet** — deferred to Drop F.2. User must SQL/service-role toggle to test for now.
- **No username modal yet** — deferred to Drop F.2. All 3 live users already have usernames from the migration backfill.

## ⚠️ Gotchas learned

1. **Public queries need `q(t, p, false)` when signed out** — `q(t, p, true)` would route through `authFetch` → `ensureSession` → try to refresh a non-existent token. Use `needsAuth=!!tk` to toggle per-call.
2. **Router fires synchronously on init, but reference data loads async** — the stub renderers fetch their own `pokemon` row per call (don't rely on `allPkmn` being loaded). Slight extra network cost is worth the decoupling.
3. **`body.pub-route` sidebar hide is opt-out via `body.pub-route-authed`** — signed-in viewers on someone else's share link keep the sidebar so they can navigate back to their own app. Signed-out visitors get sidebar hidden entirely (no auth UI, no dangling nav items).
4. **Hash navigation needs `history.replaceState` + manual handleHashRoute** — calling `location.hash=''` works but leaves `#` in the URL. `replaceState` + explicit call is cleaner.

## 📦 Deferred to Drop F.2+

- Share toggle UI in build/team editor save panel
- Share button (Web Share API + clipboard fallback)
- Full read-only public detail layout (stats hex, 4 move cards, item row, nature, ability)
- Username modal triggered when toggling public without a handle
- Signup form username + display name fields
- Like button (writes to `build_likes` / `team_likes`)
- Copy-to-my-builds (duplicates row, strips share_code + is_public + is_favourite, appends "(copied)")
- Public trainer profile page (`#/u/<username>`)


## 🐛 Bug found in initial F.1 testing: glow absolute→relative override

**Symptom**: Massive empty space at the top of the `.pub-card`, content pushed to the bottom.

**Cause**: CSS specificity collision. `.pub-card > *{position:relative;z-index:1}` (specificity 0,1,1) was declared AFTER `.pub-glow{position:absolute}` (specificity 0,1,0). Same specificity → later rule wins → glow became `position:relative` and claimed its full 280×280px in the flex flow.

**Fix**: `.pub-card > :not(.pub-glow){position:relative;z-index:1}` — excludes the glow from the universal child rule, letting it keep its `position:absolute`.

**Lesson**: when you have an absolutely-positioned descendant and want to set common props on all OTHER children, always use `:not()` rather than a universal selector. Same-specificity overrides bite.

# 🔗 Drop F.2a — Build editor share toggle (April 22 2026)

## ✅ What shipped

- **New Share card** in the build editor form. Renders ONLY when editing an existing build (`editBuildId` truthy) — drafts can't be shared.
- **Toggle switch** — `role="switch"`, keyboard-accessible (Space/Enter), standard 48×28 pill shape with red accent when ON.
- **URL row + Copy button** reveals below the toggle when ON. URL is `location.origin + location.pathname + '#/b/<code>'` so it works on localhost (Live Server), GitHub Pages, and any future custom domain without code changes.
- **Clipboard copy** via `navigator.clipboard.writeText()` with textarea+execCommand fallback for older iOS contexts. Button flashes green ✓ "Copied!" for 1.5s on success.
- **Optimistic UI with loading lock** — toggle gets `.loading` class during the API round-trip so rapid double-taps can't race. Visual state updates in place; no full editor re-render (preserves slider drag patterns per the existing playbook rule).
- **Username gate (partial)** — if `userProfile.username` is empty, turning ON shows error toast "Set a username first — coming in the next drop" and the toggle doesn't flip. All 3 current users have usernames from the migration so it never fires in practice.

## 🧪 What it does on toggle

ON (first time):
1. Call `ensureShareCode('build', editBuildId)` from `app-router.js` — returns existing code OR generates fresh 5-char base62 + PATCHes `share_code` with retry on unique collision
2. PATCH `is_public=true`
3. Update `allBuilds` local state so re-renders reflect new values
4. Reveal URL row, add red tint to card border
5. Toast "Build is now public ✨"

ON (re-enable after OFF):
- Skips code generation (reuses stored `share_code`)
- Same URL as before — existing shared links work again

OFF:
1. PATCH `is_public=false` (keeps `share_code`)
2. Hide URL row, remove red tint
3. Toast "Build is now private"

## 🔑 Key decisions

- **Card lives at the bottom of the editor form** — after Strategy (collapsible), before the save-bar. Keeps the primary build-editing flow uncluttered; sharing is a post-save action.
- **Red tint only in active state** — signals "public" (attention) without making the OFF state look disabled.
- **Prod URL shown even on localhost** — wait, actually we use `location.origin` so Live Server shows `localhost:5500/#/b/xxx`. That's correct behaviour; tester sees their actual URL.
- **No full editor re-render on toggle** — same discipline as slider oninput. DOM updates target `#edShareToggle`, `#edShareBody`, `#edShareUrl`.

## ⚠️ Gotchas

1. **`upd()` not `authFetch()` directly** — reuses the existing helper that already handles PATCH with `Prefer: return=representation` + auth refresh + error shape.
2. **`edShareSectionHtml()` returns '' for new builds** — always safe to call from `renderEditorForm()` without an if-wrapper.
3. **`share_code` preserved on OFF** — design choice, not accidental. If you want "regenerate URL on re-enable", we'd explicitly NULL it on OFF.
4. **Web Share API deferred to Drop F.2b** — just clipboard for now. Share button in build *detail* view will use `navigator.share()` with clipboard fallback.

## 📦 Deferred to Drop F.2b

- Share button in build DETAIL view (Web Share API + clipboard fallback + toast)
- Team editor share toggle (same pattern applied to `teams` table)

## 📦 Deferred to Drop F.3

- Full username modal (live availability check + validation) when toggling ON without a username set
- Signup form username + display name fields

## 🔑 Key decisions

- **View update required** — `build_details` view doesn't auto-inherit new base-table columns; had to DROP + CREATE with expanded column list. Listed rollback SQL in workspace (`build-details-view.sql`).
- **share_fields stored as NULL by default** — means "share everything". Only populated when user hides a field. Reduces DB noise.
- **Move display on public detail** — richer than in-app: BP/priority/category shown. VGC sharers want quick-scan info.
- **Strategy never shared** — explicitly excluded in renderer; no checkbox in customise panel; editor placeholder labels it `NEVER SHARED`. Notes too.
- **Type coverage for teams defaults OFF** — opt-in per team via customise panel single toggle.
- **Member card tap-through** — public members navigate to their build URL, private ones show 🔒 badge and don't react.
- **Pill pattern** — small green `🌐 Public` pill, collapsible, reveals URL + Copy + Share inline. Same component on both build and team detail pages. Only appears when is_public=true.
- **Shiny treatment** — swap to `shiny_url`, gold→red glow tint on hero card, animated ✦ sparkle points, SHINY pill badge top-right of sprite.

## ⚠️ Gotchas

1. **View columns don't propagate** — after adding a column to the base table, `build_details` VIEW must be manually rebuilt. Check `pg_get_viewdef('public.build_details')` when debugging missing columns in `loadBuilds()` output.
2. **PostgREST nested embedding syntax** — `select=col,relation(col1,col2,nested(col))` works for reading but not writing. We only use it for reads, so fine.
3. **RLS + nested embeds** — anon read policy on `team_builds` is what lets the nested fetch work when the team is public. Verified live.
4. **Share_fields default NULL strategy** — renderer's `pubGetShareFields` treats NULL as "all true" for builds, "all false" for teams (since team only has `type_coverage:false` default). Different default tables = different null semantics. Documented in the helper.
5. **Light Ice/Steel/Fairy type chips** — default white-on-gradient hurts contrast against their pale mid shades. Coverage pill renderer detects these and forces `color:#1e293b` instead.
